1.包

什么是包？

  • 包是用来分门别类的管理各种不同类的，类似于文件夹，建包利于程序的管理和维护。

2.权限修饰符

什么是权限修饰符？

  • 权限修饰符：是用来控制一个成员能够被访问的范围的。

  • 可以修饰成员变量，方法，构造器，内部类，不同权限修饰符修饰的成员能够被访问的范围将受到限制。

权限修饰符的分类和具体作用范围：

  • 权限修饰符（访问修饰符）：有四种作用范围由小到大（==private -> 缺省 -> protected - > public==）
修饰符 同一个类中 同一个包中的其他类 不同包下的子类 不同包下的无关类

学完权限修饰符需要具备如下能力

  • 能够识别别人定义的成员的访问权限。

  • 自己定义成员（方法，成员变量，构造器等）一般满足如下要求：

    • 成员变量一般私有。

    • 方法一般公开。

    • 如果该成员只希望本类访问，使用private修饰。

    • 如果该成员只希望本类，同一个包下的其他类和子类访问，使用protected修饰。

2.final关键字

final的作用

  • final关键字是最终的意思，可以修饰（==方法，变量，类==）（一般自己不用，但是有时候有，如Java的String类是final修饰）

  • 修饰方法：表示该方法是==最终方法==，不能被重写。

  • 修饰变量：表示该变量第一次赋值后，不能再次被赋值（有且仅有一次能被赋值）

  • 修饰类：表明该类是最终类，不能被继承。

final修饰变量的注意

  • final修饰的变量是基本类型：那么变量存储的数据值不能发生改变。
  • final修饰的变量是引用类型：那么变量存储的地址值不能发生改变（即，引用这一关系不能改变），但是地址指向的对象内容是可以发生变化的。

3.常量

3.1 常量概述和基本作用

常量的定义

  • 常量是使用了public static final修饰的成员变量，必须有初始化值，而且执行的过程中其值不能被改变。

  • 常量必须定义在类中，而不能在方法中。（public和static不能在方法中定义变量）

  • 常量的作用和好处：可以用于做系统的配置信息，方便程序的维护，同时也能提高可读性。

常量命名规范：

  • 英文单词全部大写，多个单词以下划线连接起来。

常量的执行原理

  • 在编译阶段会进行”宏替换”，把使用常量的地方全部替换成真实的字面量。（类似于C语言的define)
  • 这样做的好处是让使用常量的程序的执行性能与直接使用字面量是一样的。

5.抽象类

5.1 抽象类概述

抽象类

  • 某个父类知道其所有子类要完成某功能，但是每个子类的完成情况都不一样，==父类就只定义该功能的基本要求，具体实现有子类完成==，这个类就可以是一个抽象类，==抽象类其实就是一种不完全的设计图==。（类似C语言的声明方法（？））

抽象类的使用总结与注意事项

  • 抽象类是用来被继承的，抽象方法是父类==交给子类重写与实现==的。

  • 一个类如果继承了抽象类，那么这个类必须==重写完抽象类的全部抽象方法===，否则这个类也必须定义成抽象类，普通类也不能得到抽象方法。（因为如果不重写，那么就应该是抽象类继承抽象类）

  • 类似于模板，可提高代码复用。

5.3 抽象类的特征、注意事项

特征和注意事项

  • 有得有失: ==得到了抽象方法，失去了创建对象的能力==。

  • 抽象类为什么不能创建对象？

    抽象类的抽象方法没有实现，创建了调用也是无意义的。而且java创建抽象类对象是错误，不能编译

  • 类有的成员（成员变量、方法、构造器）抽象类都具备（只是不能创建对象）。

  • ==抽象类中不一定有抽象方法，有抽象方法的类一定是抽象类==。

  • ==一个类继承了抽象类必须重写完抽象类的全部抽象方法，否则这个类也必须定义成抽象类==。

  • 不能用abstract修饰变量、代码块、构造器。（只有抽象类和抽象方法的概念，别的都不谈抽象）

final和abstract是什么关系？

  • ==互斥关系==

  • abstract定义的抽象类作为模板让子类继承，final定义的类不能被继承。

  • 抽象方法定义通用功为了让子类重写，final定义的方法子类不能重写。

5.4 抽象类的应用知识：模板方法模式

什么时候使用模板方法模式

  • 使用场景说明：当系统中出现同一个功能多处在开发，而该功能中大部分代码是一样的，只有其中==部分功能可能不同==的时候。

模板方法实现步骤

  • 把功能定义成一个所谓的模板方法，放在抽象类中，==模板方法中只定义通用且能确定的代码==。

  • ==模板方法中不能决定的功能定义成抽象方法让具体子类去实现==。

  • 即，抽象父类中定义一个基本的方法写好了大部分功能，剩余不同需求的功能暂时交给抽象方法，之后子类继承此父类再对抽象方法进行重写，实现各自的剩余功能。

  • 注意事项：

    • 子类构造器得使用super传到父类。
    • ==模板方法建议使用final修饰，会更加专业==。
      • 因为模板方法是给子类直接使用的，不能让子类重写，重写了模板方法就失效了，加final为了安全，防止重写。子类该重写的是抽象方法。

6.1 接口概述和特点

接口是什么

  • 接口就是体现规范的，其中用抽象方法定义的一组行为规范，接口更加彻底的抽象。
  • 体现了现实世界中“如果你是这类事物…则必须完成某些行为…”的思想。

  • 类和接口的关系：多实现。（实现多个接口）

  • ==接口和接口的关系：多继承，一个接口可以同时继承多个接口==。

    示例：

接口多继承的作用和情况

  • 规范合并，整合多个接口为同一个接口，便于子类实现。
  • 接口多继承时，相同方法并无大碍，因为最后实现的方法都为这一个方法。
  • 当两个接口中的==方法同名但返回值类型不同==时，==规范冲突不可继承==（不可同时实现2个不同返回值的方法（矛盾））。
  • 当两个接口中的方法同名但是只是参数列表不同时，需要在实现类中实现这两个不同参数的方法。

6.4 JDK8开始后接口新增方法

JDK8版本开始之后，java只对接口的成员方法进行了新增（接口就不纯洁了）

原因如下

场景：

  1. 项目Version1.0成功上线没有问题
  2. 项目Version2.0需要对Inter接口丰富，加入10个新的抽象方法，此时改了接口就要让所有实现类实现这些方法。（那么就要去重构所有的实现类，牵一发而动全身）

那么就需要接口在丰富接口功能的同时，又不对子类代码进行更改。即此后……

允许接口中直接定义带有方法体的方法

第一种：默认方法

  • 类似之前写的普通实例方法，但是必须用default修饰(实际上就相当于是实例方法)

  • 默认会public修饰。需要用接口的实现类的对象来调用(因为接口没有对象）)

第二种：静态方法

  • 默认会public修饰，必须static修饰。(public 不用加)

  • 注意：接口的静态方法必须用本身的接口名来调用。（其它类如实现类不可调用）

第三种：私有方法

  • 就是私有的实例方法:，必须使用private修饰，从JDK 1.9才开始有的。

  • 只能在本类中被其他的默认方法或者私有方法访问。

总结

注意：JDK8新增的3种方法我们自己在开发中很少使用，通常是Java源码涉及到的，我们只需要理解、识别语法、明白调用关系即可。

6.5 使用接口的注意事项

接口的注意事项

  • 接口不能创建对象（接口更加彻底的抽象，没有构造器）

  • 一个类==实现多个接口，多个接口中有同样的静态方法不冲突==。（接口的静态方法本来实现类就不能调用，只允许接口自己调用）

  • 一个类继承了父类，同时又实现了接口，父类中和接口中有==同名方法，默认优先用父类的==。

    注意：extends要先于implements写（先有亲爸再有干爹）（实际编程几乎不会出现这种情况）

  • 一个类实现了多个接口，==多个接口中存在同名的默认方法，不冲突，这个类重写该方法即可==。

    注意：为了不让错误出现（类似于类禁止多继承方法同名错误），可以在此类中重写这一方法。（极端）

  • ==一个接口继承多个接口没有问题的==，如果==多个接口中存在规范冲突则不能多继承==。

    如：同名方法但返回值不同，就不可多继承；而同样方法则可以多继承，可以进行规范合并。

总结

接口就是：

  • 定义接口，描述功能
  • 实现对接，即一个实现，多个接口。软件解耦，多模块对接。（项目思想）
